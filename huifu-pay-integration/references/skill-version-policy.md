# Skill 版本与升级提示策略

这份文件只处理 Skill 自身版本、用户升级路径和能力缺失时的提示口径。普通支付接入、联调排查和上线检查不要读取本文件。

## 什么时候读取

- 用户询问“怎么更新 Skill”“是不是最新版”“1.4.0 有没有支持”“当前版本是多少”。
- 用户指出官网、维护者说明或新版包和本地 Skill 口径不一致。
- 用户要求当前 `1.3.1` 不覆盖的能力，并追问后续怎么获得支持。
- 用户问“为什么没有这个能力”“是否需要升级后才能做”。

不要因为普通聚合支付、托管支付、checkout-js、退款、查单、异步通知或上线检查问题主动打断用户提示升级。

## 用户如何更新

根据用户实际分发方式选择一种说法，不要编造下载地址或内部发布渠道。

| 分发方式 | 建议口径 |
| --- | --- |
| Git 仓库使用 | 从维护方仓库拉取最新版本后，重新加载或重启使用 Skill 的客户端，再确认 `huifu-pay-integration/SKILL.md` 中的版本事实 |
| 本地目录复制 | 用维护方发布的新 `huifu-pay-integration` 目录整体覆盖旧目录，不要只混合替换部分 reference |
| 内部发布包 | 下载维护方发布的新包后替换 Skill 目录；如无法确认版本，联系维护者核验 |
| 不清楚来源 | 先让用户联系 Skill 维护者确认当前分发渠道、目标版本和替换方式 |

升级前后都不要要求用户写入商户密钥、订单号、真实流水或生产配置到 Skill 包中。

## 什么时候提示升级

- 用户明确问升级方式时，直接说明当前本地版本和可执行更新路径。
- 固定说明“只有用户触发版本/升级问题时提示”，不要说成后台主动提醒。
- 用户请求当前版本不覆盖能力时，先说明 `1.3.1` 不覆盖，再建议关注维护方后续版本或转人工确认。
- 用户要求实时核对官网最新口径时，说明本地 Skill 不内置实时官网抓取；需要人工或联网核验，核验后由维护者刷新本地 reference。
- 用户只是要普通接入方案时，不主动输出升级提示。

## 输出规则

版本/升级类回答必须包含：

- 本轮实际使用的 references：通常列 `references/shared-overview.md`、`references/shared-versioning-policy.md`、`references/skill-version-policy.md`。
- 当前本地版本。
- 当前问题是否属于 `1.3.1` 已覆盖能力。
- 用户可执行的更新方式或维护者确认路径。
- 升级后必须让用户确认 `huifu-pay-integration/SKILL.md` 中的版本事实，不能只看目录名或口头版本号。
- 只有用户触发版本/升级问题时提示。
- 固定写明“不能主动联网检查或主动推送升级”，不能假装已确认最新版，不能提供未知下载地址。

## 推荐短句

```text
当前本地 Skill 版本为 1.3.1。Skill 不能主动联网检查或主动推送升级；只有用户触发版本/升级问题时提示。如果你使用的是 Git 仓库版本，请从维护方仓库拉取最新包后重新加载/重启客户端，并确认 huifu-pay-integration/SKILL.md 中的版本事实。
```

```text
你问到的能力不在 1.3.1 覆盖范围内。我可以先按当前本地 reference 完成已覆盖部分；未覆盖能力建议联系 Skill 维护者确认是否已有后续版本或需要转人工处理。
```
