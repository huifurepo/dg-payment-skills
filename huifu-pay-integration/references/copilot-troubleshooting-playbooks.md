## 输出结构

问题排查回答固定使用：

| 项 | 内容 |
| --- | --- |
| 问题判断 | 这是什么类型的问题 |
| 可能原因 | 按概率列 2-4 个原因 |
| 先检查 | 按顺序检查配置、参数、流水、状态、日志 |
| 修复动作 | 能执行的修改或补充 |
| 对存量系统影响 | 会影响哪些原逻辑 |
| 验证方式 | 怎么确认修好 |
| 升级人工 | 什么条件下停止自动判断 |

## 问题分类

| 分类 | 典型问题 |
| --- | --- |
| 方案选错 | H5 收银台却走聚合正扫 |
| 配置缺失 | 缺 `project_id`、`notify_url`、权限未开通 |
| 商户进件/实名 | 进件部分成功、未返回 `huifu_id`、需要补齐资料、实名状态延迟 |
| 接口/数据权限 | 接口权限认证失败、数据权限认证失败、无交易权限 |
| 通道/appid 配置 | 路由无可用通道、`sub_mch_id` 与 `sub_appid` 不匹配、URL 未注册 |
| SDK 问题 | Java 方法不存在、PHP `init.php` 或类加载失败 |
| 请求头问题 | 缺 `jpt-x-skill-source` 或 `jpt-x-skill-huifu_id` |
| 签名验签 | 请求签名失败、异步通知验签失败、前端拉起签名失败 |
| 参数格式 | JSON 字符串、金额、日期、分账参数错误 |
| 状态判断 | 把 `resp_code` 当交易成功 |
| 异步通知 | 收不到、重复收到、回包不正确 |
| 查单 | 查不到订单、订单处理中、原交易流水混淆 |
| 退款 | 退款申请与退款成功混淆、退款流水用错 |
| 分账 | 接收方、金额、比例、权限、手续费承担方问题 |
| checkout-js | 前端 callback 成功但服务端订单未成功 |
| 存量改造 | 不知道新增什么、改什么、保留什么 |
| 存量幂等 | 重复通知导致重复发货或重复改状态 |
| 技术支持 | 已按本地 Playbook 排查仍解决不了、需要官方 AI 技能包技术支持联系方式 |

## 高频 Playbook

| 问题 | 先检查 | 修复/结论 |
| --- | --- | --- |
| `orderExpireTime` 格式不正确 | 时间格式、是否未来时间 | 使用 `yyyyMMddHHmmss` 且必须晚于当前时间 |
| 商户进件返回成功但后续业务开通失败 | 是否返回 `huifu_id`、银行卡/结算/取现/渠道开通是否完成 | 有 `huifu_id` 只能说明账户已创建；继续补齐资料或业务开通，不把它等同交易可用 |
| 接口权限认证失败 / `20003` | `sys_id`、接口路径、环境、当前接口权限 | 不是代码字段拼写问题；整理材料让汇付/运营开通接口权限 |
| 数据权限认证失败 | `product_id`、`sys_id`、`huifu_id`、归属关系 | 修正商户/服务商层级，不猜测归属 |
| 该路由下未配置可用通道 | 微信/支付宝/银联通道、线上/线下业务、`pay_channel`/`pay_scene`、`channel_no` | 补开通道和场景；不指定渠道时不要传空字符串 `channel_no` |
| 支付宝/微信入驻信息配置有误 | 是否已入驻目标渠道、是否多渠道入驻、`pay_channel`/`pay_scene` 是否缺失 | 多渠道场景明确传渠道和场景；单渠道可按接口口径省略可选项 |
| `sub_mch_id` 与 `sub_appid` 不匹配 | appid 是否绑定商户、配置值是否有首尾空格 | 先修控台或配置接口绑定关系，不要换 SDK |
| `sub_openid` 与 `sub_appid` 不匹配 | openid 是否从同一 appid 授权流程获取 | 重新走对应 appid 的授权流程，不能跨公众号/小程序复用 |
| 重复交易 | `req_seq_id` 是否重复 | 同一 huifu_id 下当天请求流水保持唯一 |
| 收到 2 条异步消息 | `notify_type`、`trans_stat`、`acct_stat` | 区分交易异步和账务异步，处理必须幂等 |
| URL 未注册或 `redirect_uri` 域名不一致 | 微信/支付宝控台配置、授权目录/授权域名、appid、实际回跳地址、`notify_url`、`callback_url` | 在渠道控台修正对应 appid 的授权目录、网页授权域名或回跳配置；目录通常以 `/` 结尾，配置生效可能有延迟，仍失败时转运营/商务 |
| Webhook 找不到 `sign` | URL query、header、原始请求体读取方式 | FAQ 口径中 `sign` 常在 URL 参数里；验签仍用原始 body + endpoint key |
| 收不到商户交易消息 | 控台 Webhook 地址、是否统一接收下属机构事件、服务端公网可达 | 服务商接子商户事件时要开启统一接收下属机构事件配置 |
| 支付验证签名失败 | API 签名、前端拉起签名、Webhook MD5 是否混用 | 按场景选择 `shared-signing-v2.md` 或 `shared-webhook-signing.md` |
| `jpt-x-skill-source` 缺失 | SDK 是否配置 `skill_source` | Java/PHP 官方主链路优先使用 SDK 自动补头 |
| `jpt-x-skill-huifu_id` 缺失 | 请求 `data.huifu_id` 是否存在且非空 | 存在时必须与本次请求 `huifu_id` 一致 |
| 查单查不到 | 原支付 `req_date`、`req_seq_id`、`hf_seq_id` 是否来自落库 | 不临时猜流水，先核对下单落库记录 |
| 托管交易查询交易不存在 | 预下单后是否已实际拉起支付、当前 `trans_stat` 是否处理中 | 未拉起支付前查不到不一定是最终失败；处理中继续查单补偿 |
| 退款一直处理中 | 原交易状态、退款流水、是否通道或大额处理中 | 轮询退款查询，不换流水并发重试同一笔退款 |
| 请求频率过高，请于 1 分钟后重试 | 同商户/同订单是否并发退款或高频重试 | 降频并等待终态，避免重复退款请求 |
| 退款单不存在 | 退款请求流水和原交易流水是否混用 | 查询退款看退款交易自身定位键 |
| 托管项目信息不存在 | `project_id`、项目状态、托管权限 | 检查托管项目是否存在且进行中 |
| 前端 callback 成功但订单未成功 | 服务端查单/异步通知是否完成 | callback 只触发后端确认，不直接改订单成功 |
| 分账比例或金额错误 | 分账权限、接收方、金额总和、比例格式 | 修正分账串并确认权限，生产争议转人工 |
| 对账文件未生成或重新生成查不到 | 对账功能是否开通、跑批是否完成、文件日期和任务状态 | 先查配置和任务状态；文件为空或跑批中不等同接口错误 |
| DM 取现额度不足 / 发票未收到 | 取现额度、账户余额、开通状态、税务/运营流程 | 资金和发票问题转人工，不写支付接口代码解决 |
| PHP 找不到类 | SDK 版本、`init.php`、request 类显式加载 | 读取 `aggregation-php-adapter.md` 或 `hostingpay-php-adapter.md` |
| 本地排查仍解决不了 / 需要技术支持 | 已排查项、环境、接口路径、SDK 版本、脱敏请求/响应、流水、时间和影响范围 | 整理材料后给官方 AI 技能包技术支持链接：`https://paas.huifu.com/docs/devtools/#/skillsv1_0?id=support` |

## 升级人工

遇到以下情况只整理材料，不直接给结论：

- 生产故障归因、资损争议、清结算争议。
- 费率、合规、政策审批、通道准入、资质审核。
- 商户主体、渠道号、appid、项目权限无法由代码侧判断。
- 用户提供真实密钥、真实用户数据、真实商户数据。
- 官方文档、SDK 源码和实际响应存在高风险冲突。
- 用户已按本地 Playbook 排查仍无法解决，或明确要求官方 AI 技能包技术支持联系方式。

升级人工时固定整理：环境、产品线/接口、SDK 及版本、脱敏标识、脱敏请求/响应、请求/平台流水、时间线、脱敏日志、影响范围和已做动作；缺项要明确列出。再给支持入口，不要把真实密钥、完整身份证/手机号、真实 openid、完整订单号或完整生产日志写入回答。官方 AI 技能包技术支持入口：`https://paas.huifu.com/docs/devtools/#/skillsv1_0?id=support`。
