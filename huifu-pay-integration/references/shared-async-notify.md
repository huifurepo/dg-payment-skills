# 异步通知规则

这份文档只保留当前 Skill 包共用的接口 `notify_url` 异步通知规则。

控台 Webhook 是另一套平台事件订阅机制，不适用本文件的 RSA 验签和 `RECV_ORD_ID_` 回包规则；Webhook 签名看 `shared-webhook-signing.md`。

## 基本规则

- 通知方式：HTTP(S) `POST`
- 编码：`UTF-8`
- 默认超时：`5` 秒
- 失败重试：最多 `3` 次
- 不支持 HTTP 重定向
- `notify_url` 不要带查询参数
- 自定义端口建议落在 `8000-9005`

## 收到通知后的返回要求

商户侧收到通知后必须：

1. 返回 HTTP `200`
2. 响应体返回 `RECV_ORD_ID_` + 请求流水号

如果没有按这个格式回包，汇付会继续重试。

## 回调数据体

最容易混淆的点：

- 支付交易类回调，业务数据通常在 `resp_data`
- 商户配置类回调，业务数据通常在 `data`

处理时不要把 `resp_data` 和 `data` 混用。

## 回调验签

- 回调也必须验签
- 验签原文是回调里的业务数据体
- 异步验签不需要再做字段排序
- 这里说的是接口 `notify_url`，使用汇付 RSA 公钥验签
- 控台 Webhook 使用终端密钥 + MD5，不走 RSA

## 幂等要求

同一条异步通知可能会重复到达，业务侧必须做幂等。

建议至少满足：

- 建议建立事件流水表，幂等键至少包含 `huifu_id`、`req_seq_id` / `hf_seq_id`、`notify_type` / `trans_type`、`trans_stat` / `event_type`
- 订单状态更新必须再加业务订单状态锁，核心状态更新成功后才标记事件已处理
- 已处理过且业务状态已安全落地的重复通知才直接返回成功回包

## 状态确认建议

更稳妥的处理顺序是：

1. 收到异步通知先验签
2. 落原始事件并做事件级幂等判断
3. 对核心支付/退款状态调用查询接口做二次确认或补偿查询
4. 通过订单状态锁更新业务状态，成功后再标记事件已处理

特别注意：

- 前端页面回跳不能替代异步通知
- 单次异步通知也不能替代主动查询
