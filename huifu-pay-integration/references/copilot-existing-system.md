## 基本原则

1. 先理解现有系统，再给方案。
2. 优先新增配置、适配层、服务类和回调处理，不直接替换订单系统。
3. 涉及原订单状态、发货、开通权益、库存、账务的修改必须标为人工确认。
4. 支付终态不能由同步受理、前端 callback 或单次日志直接决定。
5. 退款、异步通知和查单必须幂等。

## 触发条件

满足任一条件，就按存量系统或疑似存量系统处理：

- 用户明确说已有订单系统、商城、支付服务、回调逻辑、退款逻辑、对账、告警或订单状态机。
- 用户说“第一次接汇付”，但当前项目或贴出的代码里已经有 `OrderService`、`PaymentService`、`PayController`、`NotifyController`、`RefundService`、订单状态枚举、发货/权益开通等业务链路。
- 用户只说“我的项目/当前项目/线上项目要接汇付”，且可见代码不是空项目，应先标为“疑似存量”，不要直接判定为非存量。

“第一次接汇付”只代表第一次接入汇付渠道，不代表从零新建业务系统。只有用户明确说明没有现有订单、支付、回调、退款和状态机，才按非存量处理。

## 需要识别的现状

| 信息 | 用途 |
| --- | --- |
| 现有支付入口 | 判断新增 Adapter 还是接入已有 PaymentService |
| 现有订单状态 | 做汇付状态到业务状态的映射 |
| 现有回调逻辑 | 判断是否已有验签、幂等、固定回包 |
| 现有查单逻辑 | 判断是否需要补主动查单或补单任务 |
| 现有退款逻辑 | 判断退款申请、退款查询、退款状态落库 |
| 现有落库字段 | 判断是否保存 `req_seq_id`、`req_date`、`hf_seq_id` |
| 改造目标 | 新增支付、补退款、接分账、接托管、修问题 |

## 改造建议卡

存量系统回答优先使用这张卡：

| 项 | 输出要求 |
| --- | --- |
| 建议新增 | `HuifuConfig`、`HuifuPaymentAdapter`、`HuifuNotifyHandler`、`HuifuQueryService`、`HuifuRefundService` 等按项目命名 |
| 建议保留 | 原订单创建、订单号生成、业务发货/权益/发券逻辑 |
| 需要人工确认后再改 | 订单状态映射、支付成功动作、退款成功后的库存/权益/账务、灰度路由 |
| 建议补充落库字段 | `req_seq_id`、`req_date`、`hf_seq_id`、`trans_stat` / `order_stat`、退款请求流水 |
| 风险点 | 前端 callback 直接改成功、未验签、重复通知重复发货、退款流水混用 |

## 状态映射

| 汇付信息 | 建议业务语义 |
| --- | --- |
| 同步 `resp_code` 成功 | 请求受理或接口处理成功，不等于支付终态 |
| `trans_stat=S` 或明确支付成功状态 | 可进入支付成功候选，但仍要经过幂等和订单状态保护 |
| `trans_stat=F` | 支付失败或交易失败，按业务规则处理 |
| `order_stat` 处理中 | 保持支付中，继续查单或等待异步通知 |
| 前端 callback 成功 | 只表示前端流程返回，不直接改订单成功 |
| 退款申请成功 | 只表示退款请求已受理，最终通过退款查询和异步通知闭环确认 |

## 高频场景

| 场景 | 处理 |
| --- | --- |
| 已有订单系统，新增汇付支付 | 新增汇付支付适配层和状态确认链路 |
| 已有其他支付渠道，新增汇付渠道 | 复用支付渠道扩展点，新增汇付状态映射 |
| 已接下单，现在补退款 | 直接进入退款和退款查询方案，不重走首次接入 |
| 前端 callback 成功就改订单成功 | 改成服务端闭环确认：异步通知验签/幂等 + 查单二次确认或补偿查询 |
| 重复通知导致重复处理 | 增加幂等键和订单状态保护 |
| 退款查询流水混淆 | 区分原支付流水、退款请求流水、汇付平台流水 |
| 现有订单增加分账 | 检查分账权限、接收方、金额/比例和手续费承担 |
| 商城接托管 H5/PC | 新增预下单和跳转逻辑，终态仍由服务端确认 |

## 上线回归

- 新旧渠道并存时，必须验证原渠道不受影响。
- 支付成功后的业务动作必须只执行一次。
- 重复异步通知、查单补单和人工重试不能重复发货。
- 退款成功后的库存、权益、账务处理必须有明确责任边界。
- 灰度、回滚和日志脱敏必须在上线前确认。
