# 参数、返回和代码片段检查

本文件定义用户贴请求参数、返回参数或代码片段时的评审口径。

## 检查结论等级

| 等级 | 含义 | 处理 |
| --- | --- | --- |
| 阻断 | 当前不能继续联调 | 先修复，再重试 |
| 高风险 | 可以继续但可能导致状态误判或资损 | 明确风险和修复建议 |
| 提醒 | 建议优化，不一定阻塞 | 给出改进项 |
| 通过 | 基本符合当前场景 | 给下一步联调建议 |

## 请求参数检查

重点检查：

- 必填字段是否缺失。
- `req_date`、`req_seq_id`、`huifu_id`、金额、日期格式是否正确。
- `notify_url` 与 `callback_url` 是否混用。
- 托管支付是否缺 `project_id` 或托管项目配置。
- checkout-js 是否缺服务端托管预下单结果契约。
- 分账、补贴、实名、大额、终端信息是否放对层级。
- JSON 字符串字段是否先按对象校验，再在 SDK 边界序列化。
- 是否把密钥、私钥、真实用户标识放到前端或示例常量。

## 返回参数解释

| 返回/状态 | 解释 |
| --- | --- |
| `resp_code` 成功 | 接口处理成功或受理成功，不等于交易终态 |
| `trans_stat` | 聚合/交易维度的最终状态判断重点 |
| `order_stat` | 托管订单状态判断重点 |
| `hf_seq_id` | 平台流水，常用于幂等、查单、日志关联 |
| `jump_url` | 托管 H5/PC 跳转入口，仍需服务端最终确认 |
| 前端 callback | 页面流程回调，不是支付成功凭证 |
| 退款同步成功 | 退款申请已受理，不等于退款成功 |

## 代码片段检查

如果用户贴现有代码，先做脱敏前置检查：发现明文私钥、公钥、真实商户号、真实订单号、openid、手机号、身份证号等敏感值时，先要求脱敏，并且不要在回答中复述完整值。

代码片段评审不直接整段重写，优先输出：

| 项 | 说明 |
| --- | --- |
| 建议保留 | 原订单创建、订单号生成、业务动作 |
| 建议新增 | 汇付请求流水落库、验签、幂等、查单兜底、退款查询 |
| 建议修改但需人工确认 | 状态流转、发货/权益、退款回滚、灰度路由 |
| 风险点 | callback 改成功、无验签、流水未落库、重复通知重复执行 |
| 验证方式 | 单元测试、联调用例、重复通知、查单补偿、退款查询 |

## 敏感信息处理

- 发现私钥、公钥、真实商户号、真实订单号、openid、手机号、身份证号时，先提醒脱敏。
- 不在回答中复述完整敏感值。
- 不把真实密钥写入代码常量、前端脚本、README 或测试 Prompt。
- 只使用占位符，例如 `<huifu_id>`、`<req_seq_id>`、`<notify_url>`。
