本文件定义聚合支付、托管支付、checkout-js 和存量增量接入的选择规则。

## 推荐规则

| 用户场景 | 推荐方案 | 关键理由 |
| --- | --- | --- |
| 标准服务端收款、二维码、付款码、小程序支付 | 聚合支付 | 接入轻量，服务端直接下单和确认状态 |
| H5/PC 收银台、页面跳转、项目制收款 | 托管支付 | 需要 `project_id`、托管项目和收银台闭环 |
| 商户自有页面嵌入支付组件或按钮 | checkout-js | 前端组件依赖托管预下单，最终状态仍回服务端确认 |
| 已经能下单，只想查单、退款、关单 | 局部能力接入 | 不重新走首次接入 |
| 已有订单系统，要新增汇付渠道 | 存量系统增量接入 | 新增适配层，不重写订单系统 |
| 当前项目已有订单、支付服务、回调、退款或状态机代码，要接汇付 | 存量业务系统首次接入汇付 | 第一次接汇付不等于新建空项目，先评估现有业务链路影响 |
| 已有汇付接入，要补退款、分账、checkout-js | 局部增量接入 | 只补当前能力和相关回归 |
| C# / Go / Rust / Node / .NET 等非 Java/PHP/Python 栈要完整代码 | 协议级说明 | 当前仓库不提供这些语言的完整模板，不能自造 |

## 不推荐规则

- 用户要 H5/PC 收银台时，不推荐走聚合支付正扫冒充收银台。
- 用户只接 checkout-js 前端时，不推荐跳过托管预下单和最终查单/异步通知。
- 用户已有订单系统时，不推荐生成全新订单项目或替换原状态机。
- 当前项目已有订单、支付、回调、退款或状态机代码时，不推荐按新建空项目输出首次接入模板。
- 用户只排查退款或查单时，不推荐回到完整首次接入链路。
- 用户缺真实生产参数时，不推荐输出“可直接联调/上线”的完整代码。

## 输出要求

方案选择输出必须包含：

1. 推荐方案。
2. 为什么推荐。
3. 为什么暂不推荐其他方案。
4. 前置配置。
5. 联调顺序。
6. 对存量系统的新增、保留、人工确认点。
7. 需要升级人工的条件。

## 产品线关系

```text
聚合支付
  -> 服务端直接下单、查单、退款、对账

托管支付
  -> 项目制预下单、H5/PC 收银台、小程序预下单、查单、退款

checkout-js
  -> 商户自有页面组件
  -> 必须依赖托管预下单
  -> 最终状态必须回到服务端闭环确认：异步通知验签/幂等 + 查单二次确认或补偿查询
```

## 能力边界

| 边界 | 处理 |
| --- | --- |
| 费率、商务、合规、政策、通道准入 | 不给结论，提示联系商务/运营/官方支持 |
| 生产事故归因 | 不定责，只整理升级人工材料 |
| 绕过验签、绕过风控、伪造支付成功 | 拒绝，并给正确验签、幂等、查单方案 |
| 非 release 能力 | 不编造，说明当前 Skill 未覆盖 |
| 真实密钥或真实用户数据 | 不保存、不打印、不写入示例 |
| C# / Go / Rust / Node / .NET 等非 Java/PHP/Python 栈完整模板 | 当前只给协议级路径和异步回调参考，不补自造模板 |
