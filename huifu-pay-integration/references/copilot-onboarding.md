## 用户类型判断

首次接入开发者通常缺少产品线选择、配置清单、联调步骤和状态闭环。优先输出接入判断卡和方案卡。

存量开发者通常已有订单、支付渠道、回调、退款、对账或告警。优先读取 `copilot-existing-system.md`，输出新增、保留、需人工确认和风险点，不直接重写原系统。

“第一次接汇付”只说明第一次接入汇付产品，不等于用户在新建空项目。只要用户表达、当前工作区或贴出的代码显示已有订单、支付入口、回调、退款、查单、发货/权益开通、订单状态枚举或状态机，就按“存量业务系统首次接入汇付”处理。

如果证据不足，不要直接判定“不是存量项目”。接入判断卡里的“是否存量”应写成“未确认”或“疑似存量”，并说明需要确认是否已有订单/支付/回调/退款/状态机；只有用户明确说从零新建且没有这些业务链路时，才写“否”。

如果用户只问某个接口字段或错误，允许跳过完整向导，直接进入 `copilot-parameter-review.md` 或 `copilot-troubleshooting-playbooks.md`。

## 接入判断卡

输出接入判断卡时固定覆盖：

| 项 | 要求 |
| --- | --- |
| 当前判断 | 用户类型、产品线、阶段、技术栈、端形态、目标 |
| 推荐方案 | 只给最匹配的一条主方案；比较需求明确时才并列 |
| 为什么 | 解释选这条路和暂不选其他路的原因 |
| 先读这些 | 3-5 份本地 reference |
| 还需要确认 | 只列会阻塞落地的真实配置或业务决策 |

## 最少澄清

只有继续回答会走错主链路时才追问：

- 无法区分聚合支付、托管支付、checkout-js。
- 无法区分服务端接入、前端页面接入、最终状态确认。
- 用户要求可联调或生产可用代码，但缺少真实 `project_id`、`notify_url`、`sys_id`、商户号、渠道标识。
- 用户只负责 checkout-js 前端，但未确认托管预下单、异步通知验签/幂等和查单补偿闭环。
- 当前技术栈没有模板，且存在多条回退路径。

## 检查点口径

硬检查点以主 `SKILL.md` 的“检查点机制”为准，本节“最少澄清”只是摘要。触发硬检查点时必须覆盖以下 6 类口径：

- 无法区分聚合支付、托管支付、checkout-js 三条主链路。
- 无法区分服务端接入、前端页面接入、最终状态确认。
- 用户要求现成代码，但当前组合无模板且存在多条回退路径。
- 用户只推进 checkout-js 前端，但未确认托管预下单、异步通知验签/幂等和查单补偿闭环。
- 用户要求可直接联调、可上线或生产可用代码，但缺少关键真实值。
- 本地 SDK 源码与文档口径冲突，且冲突点属于请求头、签名、版本或能力覆盖。

## 默认路由

| 用户表达 | 默认判断 | 入口 |
| --- | --- | --- |
| 第一次接汇付，不知道选什么 | 首次接入，先做产品线选择 | `shared-overview.md`、`copilot-solution-selection.md` |
| 标准服务端收款、扫码、付款码、小程序支付 | 聚合支付 | `aggregation-quickstart.md` |
| H5/PC 收银台、project_id、托管项目 | 托管支付 | `hostingpay-quickstart.md` |
| 自有页面嵌入 checkout 或支付按钮 | checkout-js，依赖托管预下单 | `checkout-js.md`、`checkout-js-callback-and-confirmation.md` |
| 已有系统或当前项目已有订单/支付/回调代码，新增汇付支付 | 存量增量接入 | `copilot-existing-system.md` |
| 贴了请求、返回、错误码 | 参数/问题排查 | `copilot-parameter-review.md`、`copilot-troubleshooting-playbooks.md` |
| 准备上线 | 上线检查 | `copilot-go-live-checklist.md` |
