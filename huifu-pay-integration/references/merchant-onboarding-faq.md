## 什么时候读

用户问题命中以下任一类时，优先把本文件与 `copilot-troubleshooting-playbooks.md` 组合使用：

- 商户进件是否成功、是否还能继续开通业务、是否需要重新进件。
- `接口权限认证失败`、`数据权限认证失败`、无交易权限、未开通默认账户或未开通某支付产品。
- 微信/支付宝/银联通道不可用、路由无可用通道、线上/线下场景或费率类型不匹配。
- `sub_mch_id`、`sub_appid`、`sub_openid`、授权目录、`redirect_uri`、支付签名验证失败。
- Webhook 找不到 `sign`、收不到下属机构事件、交易成功收到多条异步。
- 退款处理中、退款频率过高、分账金额/比例错误、对账文件未生成、DM 取现、发票等非代码侧问题。

## 商户进件结果判断

| 现象 | 判断 | 处理 |
| --- | --- | --- |
| 进件同步返回成功码并带 `huifu_id` | 可视为账户已创建，但不代表业务开通全部成功 | 继续检查银行卡、结算、取现、渠道业务开通、`token_no` 等后续节点 |
| 返回有 `huifu_id` 但业务开通失败 | 属于部分成功 | 通过修改/补充资料接口继续补齐，或按业务规则重新进件并废弃旧主体 |
| 未返回 `huifu_id` | 进件失败 | 不要继续拼交易请求；先补齐资料后重新进件 |
| 用户只问“接口怎么进件” | 先区分企业/个体/个人材料 | 有营业执照按企业或个体基础进件，无营业执照按个人场景；进件后仍要开通支付产品 |

不要把“进件成功”直接写成“微信/支付宝/银联交易可用”。交易可用还依赖支付产品、渠道、费率、appid 绑定、实名和风控状态。
主体名称、姓名、地址、appid 等配置类字段要去首尾空格并避免特殊字符；企业、个体、个人主体类型和证照/银行卡/手机号要匹配。

## 接口权限和数据权限

| 报错/现象 | 先检查 | 结论 |
| --- | --- | --- |
| `接口权限认证失败` 或 `20003` | `sys_id` 是否开通当前接口权限 | 这不是代码字段拼写问题；整理 `sys_id`、接口路径、环境、请求时间给汇付/运营开权限 |
| `数据权限认证失败` | `product_id`、`sys_id`、`huifu_id`、`upper_huifu_id`、服务商/子商户层级 | 不猜测归属关系；核对公共参数、请求头和主体层级 |
| 无交易权限、上级未开通、渠道商禁止开商户 | 上级服务商能力、当前商户能力、产品权限 | 代码侧只能暴露缺口；权限/准入结论转运营或商务 |
| 默认商户或默认账户配置错误 | 默认商户、账户号、业务账户是否启用 | 不自动填默认账户；向客户索取控台配置或转人工 |

排查权限类问题时，先确认请求是否真的打到目标环境；同一套 `sys_id/product_id/huifu_id` 不应在测试、生产或不同服务商层级之间混用。
接口权限、产品权限或上级链路权限需要由运营/汇付侧开通；若开通动作伴随密钥或环境配置变更，生产代码和密钥管理必须同步确认，不能由 Skill 自动替换。

## 通道和路由配置

| 报错/现象 | 先检查 | 修复口径 |
| --- | --- | --- |
| 该路由下未配置可用通道 | 微信/支付宝/银联通道是否开通，线上/线下业务是否开通 | 先补业务开通和通道配置，再重试交易 |
| `channel_no` 相关路由失败 | 是否传了空字符串 | 不需要指定通道时直接不传；不要传空值让系统匹配空通道 |
| 已入驻多个渠道但交易仍失败 | `pay_channel`、`pay_scene`、线上/线下 `fee_type` 是否匹配 | 多渠道入驻场景应明确渠道和场景；线上/线下不能混传 |
| 微信商户配置提示未入驻 | 请求场景与已开通费率类型不一致 | 按实际已开通的线上或线下类型传值，或补开对应费率 |

通道路由失败通常不是 SDK 兼容性问题。回答时先输出配置检查顺序，不要直接建议更换接口或重写 SDK。

## appid、openid 和授权目录

| 现象 | 先检查 | 修复口径 |
| --- | --- | --- |
| `sub_mch_id` 与 `sub_appid` 不匹配 | appid 是否已绑定到该商户，配置值是否有空格或历史值 | 在控台或配置接口完成绑定，清理首尾空格后重新配置 |
| `sub_appid` 与 `sub_openid` 不匹配 | `sub_openid` 是否从同一个 `sub_appid` 的授权流程获取 | 不能跨公众号/小程序复用 openid |
| `openid` 和 `sub_openid` 为空 | 授权流程是否实际返回用户标识 | 前端或小程序先完成授权；服务端不要伪造 |
| URL 未注册 | 微信支付授权目录是否配置 | 授权目录通常要求以 `/` 结尾，数量有限，配置后可能延迟生效 |
| `redirect_uri` 域名不一致 | 微信网页授权域名是否配置为当前回跳域名 | 修改渠道侧授权域名；顶级域名和完整目录匹配规则要区分 |
| 前端支付签名验证失败 | API RSA、前端拉起签名、Webhook MD5 是否混用 | 按场景选择签名材料；不要用 Webhook 终端密钥做前端支付签名 |

appid/openid 类问题必须把“配置绑定”和“运行时授权值来源”分开。配置对了但用户标识来自别的 appid，交易仍会失败。

## 实名认证、入驻和控台状态

| 现象 | 判断 | 处理 |
| --- | --- | --- |
| 提示需要补齐资料 | 微信/支付宝实名或商户认证未完成 | 补齐主体资料、图片、证照或走对应实名接口 |
| 实名状态显示不实时 | 控台列表或同步状态可能延迟 | 以详细页或官方实时查询为准；延迟类问题不要反复发交易 |
| 控台实名认证提交失败 | 图片/证件/主体信息不满足渠道要求 | 整理失败原因和材料，转运营或渠道侧处理 |
| 已入驻全部通道但仍失败 | 可能是交易场景、费率类型、appid、风控或权限未匹配 | 按通道、场景、appid、权限逐项排查，不只看“已入驻”字样 |

实名、资料审核、风控冻结和渠道准入不属于代码可自动修复范围。Skill 只整理材料和排查路径，不给审核通过或可上线承诺。

## Webhook 和异步事件配置

| 问题 | 处理口径 |
| --- | --- |
| Webhook 找不到 `sign` | FAQ 口径中 `sign` 常在请求 URL 查询参数里，不一定在 header；实现要同时按框架能力读取 query 和原始 body |
| 控台配置了服务商地址但收不到子商户事件 | 检查是否开启“统一接收下属机构 Webhook 事件”一类配置 |
| 收到 2 条交易相关异步 | 区分交易异步、账务异步和 Webhook 事件类型，业务处理必须幂等 |
| `notify_url` 与控台 Webhook 混淆 | `notify_url` 走接口异步和 RSA 验签；控台 Webhook 走原始 body + endpoint key 的 MD5 |

异步类回答必须强调：先验签，再解析，再幂等，再通过查单或补偿查询确认最终状态。

## 退款、分账、对账和资金运营边界

| 现象 | 判断 | 处理 |
| --- | --- | --- |
| 退款一直处理中 | 可能是通道或大额退款处理中 | 轮询退款查询，不换流水并发重试同一笔退款 |
| 请求频率过高 | 同一商户/订单退款请求过密 | 降频，等待终态后再发起下一步 |
| 退款已明确失败且平台未入库 | 上游限流或未落库场景 | 可按业务幂等规则重新发起新退款并使用新的请求流水；不要与“处理中”场景混用 |
| 退款单不存在 | 原支付流水、退款请求流水或平台流水混用 | 退款查询使用退款交易自身定位键 |
| 延时分账退款 | 未确认部分和已确认部分路径不同 | 未确认部分可按交易退款；已确认部分先做确认退款或资金回退后再退原付款方 |
| 分账金额总和必须等于交易金额 | 实时分账要求一次性分完 | 修正分账金额合计、接收方和手续费承担关系 |
| 分账比例格式错误或比例过高 | 比例格式或核准比例不匹配 | 修正格式；超过核准比例时转人工确认权限 |
| 对账文件未生成或查不到 | 可能未开通对账配置、跑批未完成、文件为空或重生成未完成 | 检查对账功能开通、文件日期、跑批状态和任务状态 |
| DM 取现额度不足、D1/DM 未开通、发票未收到 | 资金、结算、税务或运营问题 | 不写支付代码解决；整理商户、时间、金额、流水和影响范围转人工 |

## 输出边界

- 可以输出排查顺序、字段来源、需要客户补齐的配置和升级人工材料。
- 不输出通道准入、费率、风控、实名审核、取现额度、发票结果的最终结论。
- 不要求用户把真实密钥、真实身份证、银行卡、真实订单号或生产流水发给模型。
- 不把 FAQ 原文、截图、base64 或完整问答库沉淀进 Skill。
